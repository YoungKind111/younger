# 线索分配系统 - 产品用例文档

## 一、核心概念定义

| 术语 | 定义 |
|-----|------|
| **渠道** | **推广员-平台** 的组合（如：张三-小红书），是线索分配的最小独立单元 |
| **待分配队列** | 某渠道内尚未开始分配的线索池，按进入时间排队（先进先出） |
| **人工池** | 某渠道内所有销售都超时后进入的线索池，**必须人工处理，不再自动流转** |
| **流转中** | 线索已分配给某销售，等待确认或超时的状态 |
| **销售状态池** | **全局共享**，记录每个销售当前是"空闲"还是"待确认" |
| **分配锁定** | 销售处于"待确认"状态时，**所有渠道**的新线索都会跳过该销售 |
| **领取链接** | 企微通知中的确认链接，**单次有效，超时即失效** |
| **兜底管理员** | 渠道配置的管理人员，当线索进入人工池时接收通知并处理 |
| **优先分配资格** | 销售反馈线索无效后获得的资格，可累积，新线索到达时优先分配给该销售 |

---

## 二、渠道模型

### 2.1 渠道命名格式
```
{推广员}-{平台}

示例：
├── 渠道001：张三-小红书
├── 渠道002：张三-抖音  
├── 渠道003：李四-小红书
├── 渠道004：李四-百度
└── 渠道005：王五-线下展会
```

### 2.2 渠道特性
- 每个渠道**完全独立**（独立规则、独立队列、独立人工池）
- 推广员和平台仅作为渠道的**属性**，用于统计和筛选
- 销售状态**全局共享**（某销售在渠道A待确认时，渠道B也会跳过他）
- 每个渠道需配置**兜底管理员**（1~N人），负责处理人工池线索

## 三、用例场景

### 3.1 基础分配流程

#### UC-01 首位销售成功领取（Happy Path）

```
触发条件：新线索进入某渠道待分配队列
前置条件：该渠道销售队列首位销售处于"空闲"状态
执行流程：
  1. 系统将线索分配给首位销售（如销售A）
  2. 发送企微通知给A（含唯一领取链接，有效期X分钟，按渠道配置）
  3. A在5分钟内点击链接确认领取
  4. 线索归属A，分配完成
终态：✅ 分配成功
```

#### UC-02 首位超时，次位成功领取

```
执行流程：
  1. 线索分配给销售A，发送通知（含唯一领取链接，Token有效期X分钟，按渠道配置）
  2. A超过5分钟未确认，系统触发超时流转
     ├─ 标记A的领取链接为【已失效】
     ├─ 流转给销售B，发送新通知（新链接，新Token）
  3. 若A在超时后点击原链接，提示："该线索已超时，已分配给其他销售"
  4. B在5分钟内确认领取
终态：✅ 分配成功（归属B）
```

#### UC-03 中间顺位超时流转

```
执行流程：A超时（链接失效）→ B超时（链接失效）→ C确认领取（当前有效）
终态：✅ 分配成功（归属C）

注意：每个销售收到的领取链接都是唯一的，且仅在当前分配环节有效
```

#### UC-04 末位超时进入人工池（兜底策略）【已更新】

```
执行流程：
  1. 渠道销售队列 A→B→C 依次超时（每人X分钟）
  2. 所有链接均已失效
  3. 系统将该线索移入该渠道的【人工池】
  4. 发送通知给该渠道配置的【兜底管理员】（1~多人）：
     
     【企微通知内容】
     ─────────────────────
     🔔 线索分配提醒
     
     渠道[张三-小红书]有线索无人认领，已进入人工池
     
     线索ID：CL202403280001
     客户电话：138****8888
     来源平台：小红书
     推广员：张三
     流转历史：销售A(超时) → 销售B(超时) → 销售C(超时)
     进入人工池时间：10:15
     
     [查看详情]  [立即分配]
     ─────────────────────
     
  5. 该线索【停止自动流转】，等待管理员手动处理
终态：⚠️ 进入人工池
```

#### UC-05 人工池线索手动分配【已更新】

```
触发条件：管理员在人工池中选择线索，指定销售
执行流程：
  1. 管理员选择目标销售（可以是任意销售，不受原队列限制）
  2. 系统直接将线索分配给该销售
  3. 【立即归属，无需确认】
  4. 通知该销售："管理员已分配线索XXX给你"
  5. 可选：通知兜底管理员"线索已被XXX分配"
终态：✅ 直接分配成功
```

### 3.2 并发与排队场景

#### UC-06 销售待确认时跳过（全局生效）

```
场景：销售A正在确认渠道001的线索X，此时：
      - 渠道001来新线索Y
      - 渠道002来新线索Z
      
执行流程：
  1. 系统检测到A状态 = "待确认"
  2. 线索Y跳过A，直接查询B的状态
  3. 线索Z也跳过A，直接查询B的状态
  
规则：销售状态是全局的，一旦某销售在处理任何渠道的线索，其他渠道也跳过他
```

#### UC-07 多线索同时到达同一渠道

```
场景：渠道001（张三-小红书）同时进入3条线索

执行流程：
  1. 线索1分配给A（A变为待确认）
  2. 线索2检测到A待确认 → 跳过A → 分配给B（B变为待确认）
  3. 线索3检测到A、B都待确认 → 分配给C
  
注意：如果此时A、B、C都在待确认状态，线索3直接进入人工池（不等待）
```

#### UC-08 多线索同时到达不同渠道

```
场景：
  - 渠道001（张三-小红书）进入线索X
  - 渠道002（张三-抖音）进入线索Y
  - 渠道003（李四-小红书）进入线索Z
  
执行流程：
  三条线索【并行处理】，各自读取本渠道的销售队列
  
边界处理：
  若X和Y同时抢到同一销售A，数据库乐观锁保证一个成功，另一个顺延给B
```

#### UC-09 全渠道销售都在忙碌

```
场景：销售A/B/C都在待确认状态（处理各渠道的线索），新线索进入某渠道

执行流程：
  1. 系统遍历该渠道销售队列：A待确认→跳过，B待确认→跳过，C待确认→跳过
  2. 无空闲销售
  3. 新线索直接进入该渠道的【人工池】
  
规则：【不等待，不排队】，直接进人工池
```

### 3.3 优先分配场景（销售反馈无效线索）

#### UC-24 销售反馈无效线索获得优先资格

```
触发条件：销售在"我的线索"页面反馈某线索为"无效"
前置条件：销售已领取该线索并跟进
执行流程：
  1. 销售提交反馈：
     ├─ 选择"无效"
     ├─ 填写无效原因（空号/错号/无意向/重复/其他）
     ├─ 上传证据（录音/截图/文字说明）
     └─ 提交反馈
  2. 系统记录反馈信息
  3. 为该销售增加【1个优先分配资格】
  4. 通知销售："反馈已提交，获得优先分配资格"
     
终态：✅ 获得优先资格

规则：
  - 优先资格可累积（反馈N个无效 = N个优先资格）
  - 优先资格无有效期
  - 优先资格与销售ID绑定，跨渠道生效
```

#### UC-25 优先分配资格使用（销售空闲）

```
触发条件：新线索进入渠道，且某销售拥有优先分配资格
前置条件：该销售处于"空闲"状态
执行流程：
  1. 系统检测到销售A拥有优先分配资格
  2. 检查销售A状态 = "空闲"
  3. 【临时置顶】将线索优先分配给销售A
  4. 发送【优先分配通知】给销售A：
     
     【企微通知内容】
     ─────────────────────
     ⭐ 优先分配 - 新线索待领取
     
     您有一条优先分配的新线索！
     （因您之前反馈无效线索获得的优先资格）
     
     线索ID：CL202403300001
     渠道来源：张三-小红书
     客户姓名：李女士
     客户电话：13812348888
     
     ⭐ 优先资格信息
     当前剩余优先资格：2个
     本次分配：优先置顶
     
     ⏰ 请在 X分钟 内点击确认领取
     
     [确认领取]  [查看详情]
     ─────────────────────
     
  5. 销售A在X分钟内确认领取
  6. 分配成功，【消耗1个优先资格】
  
终态：✅ 优先分配成功，资格消耗

规则：
  - 优先分配仍需确认领取，不是直接归属
  - 多个销售有优先资格时，按获得时间先后排序
  - 确认领取后优先资格才消耗
```

#### UC-26 优先分配资格使用（销售忙碌）

```
触发条件：新线索进入渠道，且某销售拥有优先分配资格
前置条件：该销售处于"待确认"状态（正在处理其他线索）
执行流程：
  1. 系统检测到销售A拥有优先分配资格
  2. 检查销售A状态 = "待确认"（忙碌）
  3. 【跳过销售A】，检查其他有优先资格的销售
  4. 若销售B有优先资格且空闲 → 分配给销售B
  5. 若无其他有优先资格的销售 → 按原销售队列分配
  
终态：⚠️ 优先资格保留，本次未使用

规则：
  - 销售忙碌时，优先资格不消耗
  - 资格保留，等待下次新线索分配
```

#### UC-27 优先分配超时处理

```
触发条件：优先分配的线索超时未确认
执行流程：
  1. 销售A收到优先分配通知
  2. 超过X分钟未确认领取
  3. 系统触发超时流转：
     ├─ 线索流转给下一位销售（正常队列顺序）
     ├─ 销售A的【优先资格保留】（不消耗）
     └─ 发送通知给销售A："优先分配已超时，资格保留"
  4. 下一位销售接收通知并处理
  
终态：⚠️ 线索流转，资格保留

规则：
  - 优先分配超时，资格不消耗
  - 销售可继续保留优先资格，等待下次分配
```

#### UC-28 多个优先资格累积使用

```
场景：销售A反馈了3个无效线索，拥有3个优先资格

执行流程：
  第1条新线索到达：
    ├─ 销售A空闲，优先分配
    ├─ 销售A确认领取
    └─ 剩余优先资格：2个
    
  第2条新线索到达：
    ├─ 销售A空闲，优先分配
    ├─ 销售A确认领取
    └─ 剩余优先资格：1个
    
  第3条新线索到达：
    ├─ 销售A空闲，优先分配
    ├─ 销售A确认领取
    └─ 剩余优先资格：0个
    
  第4条新线索到达：
    └─ 销售A无优先资格，按原队列分配
    
终态：✅ 优先资格按先进先出顺序消耗

规则：
  - 多个优先资格按获得时间顺序使用
  - 每次成功领取消耗1个资格
  - 资格用完后续分配恢复正常队列
```

### 3.4 渠道管理场景

#### UC-10 调整渠道销售顺序

```
场景：管理员调整渠道001的销售顺序 A→B→C 改为 C→B→A

生效规则：
  - 正在流转中的线索（已分配给A待确认）：【保持原顺序走完】（A→B→C）
  - 新进入的线索：【按新顺序分配】（C→B→A）
```

#### UC-11 从渠道移除销售人员

```
场景：将销售B从渠道001的销售队列中移除

影响范围：
  - 新线索：不再分配给B
  - 正在流转中的线索（当前在A，下一个是B）：【继续分配给B】，B仍可领取
  
原因：保证线索流转的连续性，避免规则变更导致线索卡住
```

#### UC-12 停用/删除渠道

```
场景：停用渠道001（张三-小红书）

处理规则：
  - 待分配队列中的线索：【移入人工池】
  - 正在流转中的线索：【继续完成】（不受停用影响）
  - 新线索：拒绝进入，提示"渠道已停用"
```

#### UC-23 配置渠道兜底管理员【新增】

```
场景：为渠道配置兜底管理员

执行流程：
  1. 进入渠道配置页
  2. 在"兜底通知"区域选择管理员（从企微成员中选择，可多选）
  3. 保存配置
  
规则：
  - 每个渠道必须配置至少1个兜底管理员
  - 管理员可以不是销售队列成员（如主管、运营）
  - 线索进入人工池时，同时通知所有配置的管理员
```

### 3.4 跨渠道场景

#### UC-13 同推广员多渠道同时来线索

```
场景：推广员张三的小红书渠道和抖音渠道同时来线索

处理：
  - 渠道001（张三-小红书）：按渠道001规则分配
  - 渠道002（张三-抖音）：按渠道002规则分配
  
两渠道完全独立，只是推广员属性相同，用于后续统计汇总
```

#### UC-14 同平台不同推广员来线索

```
场景：张三-小红书 和 李四-小红书 同时来线索

处理：
  - 渠道001（张三-小红书）：分配给A/B/C
  - 渠道003（李四-小红书）：分配给D/E/F
  
不同渠道，不同销售队列，互不干扰
```

### 3.5 异常场景

#### UC-15 销售账号失效

```
场景：线索流转到销售B时，发现B账号被禁用

处理：
  1. 系统检测到B账号异常
  2. 记录异常日志
  3. 【直接跳过B】，流转给C
  4. 发送告警通知管理员："销售B账号异常，渠道XXX已跳过"
```

#### UC-16 企微通知失败

```
场景：调用企微接口发送通知失败

处理：
  1. 重试3次，间隔10秒
  2. 3次仍失败：视为"已通知"，按正常超时逻辑处理（X分钟后流转）
  3. 记录失败日志，供排查
```

## 四、关键业务规则汇总

| 规则编号 | 规则描述 |
|---------|---------|
| **BR-01** | 渠道 = **推广员-平台**（如：张三-小红书），每个组合是一个独立渠道 |
| **BR-02** | 每个渠道有【独立的】销售队列、待分配队列、人工池 |
| **BR-03** | 销售状态（空闲/待确认）是【全局共享】的，跨渠道生效 |
| **BR-04** | 每个渠道独立配置超时时间（X分钟） |
| **BR-05** | 销售同时只能处理**1条**待确认线索，新线索（所有渠道）自动跳过 |
| **BR-06** | 末位超时后进入【该渠道的人工池】，【不再自动流转】 |
| **BR-07** | 人工池线索手动分配后【直接归属，无需确认】 |
| **BR-08** | 每个渠道必须配置【至少1个兜底管理员】，用于接收人工池通知 |
| **BR-09** | 领取链接【单次有效，超时即失效】；线索流转给下一位后，前序销售链接自动作废 |
| **BR-10** | 线索录入时查重，重复则阻止录入并通知原销售 |
| **BR-11** | 销售反馈线索无效后，获得【优先分配资格】，可累积多个 |
| **BR-12** | 优先分配资格【无有效期】，直到成功领取线索后才消耗 |
| **BR-13** | 优先分配【仍需确认领取】，不是直接归属 |
| **BR-14** | 优先分配超时后，【资格保留】，线索正常流转给下一位 |
| **BR-15** | 管理员【不能强制回收】已分配但未确认的线索 |
| **BR-16** | 调整销售顺序【下条线索生效】，当前流转中的线索保持原顺序 |
| **BR-17** | 从渠道移除销售时，该渠道正在流转中的线索【继续完成】 |
| **BR-18** | 账号异常的销售【自动跳过】，并告警管理员 |
| **BR-19** | 全渠道销售都忙碌时，新线索【直接进入该渠道人工池】（不等待） |
| **BR-20** | 分配历史数据【永久保留】 |
| **BR-21** | 销售点击失效链接时，提示具体原因："已超时"或"已分配给他人" |
| **BR-22** | 线索进入人工池时，【同时通知】所有配置的管理员（1~N人） |
| **BR-23** | 管理员通知内容包含：线索基本信息、流转历史、快捷操作按钮 |
| **BR-24** | 管理员可以是非销售队列成员（如主管、运营人员） |
